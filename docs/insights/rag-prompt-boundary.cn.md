# RAG Prompt 推理边界：严格模式 vs. 合理推断

**项目**：VedaAide.NET  
**涉及文件**：`src/Veda.Services/QueryService.cs` → `BuildSystemPrompt()`  
**讨论日期**：2026-03-20

---

## 背景

RAG 的经典做法是在 System Prompt 中明确要求：
> "只能基于文档内容回答，否则回答'没有相关信息'。"

这样做的出发点是防止 LLM 凭空编造（幻觉）。然而在个人助理场景下，实际遇到了以下问题：

用户记录了两条笔记：
- `[2026-03-20] 我家每天晚上都要吃一把青菜`
- `[2026-03-20] 昨天中午买了三把青菜`

当用户问"明天如果我不买青菜，明天晚上还有青菜吃吗"时，严格模式的回答是：
> "I don't have enough information in the provided documents."

这个回答技术上正确（文档里没有明确写"明天有没有青菜"），但对用户而言完全没有价值——答案其实就在两条笔记的组合推算里。

---

## 核心矛盾

**严格模式的隐含假设**：知识库内容足够完整，问题可以在文档里直接找到答案。

**个人助理场景的现实**：用户记录的是**碎片化事实**，不是"完整的问答对"。知识库永远不完整的，意义在于辅助推断，而非百科全书式直接查找。

更重要的一点：**严格禁止推理并不真正减少幻觉**。LLM 该幻觉时还是会幻觉，多了一层"我不知道"的包装反而掩盖了真正的风险——用户不知道模型是"真的没有相关资料"还是"有资料但不肯推断"。

---

## 解决方案：按推理依据划线，而非"一刀禁推理"

| 情形 | 处理方式 |
|---|---|
| 答案可从 context 直接读出 | 直接回答 |
| 答案需跨多个 context 片段 + 常识推算 | **允许**，但在回答中注明推断依据 |
| 答案需要 context 以外的世界知识 | 拒绝，或明确标注"超出笔记范围" |
| Context 完全没有相关内容 | 明确拒绝 |

反映在 `BuildSystemPrompt()` 的规则中：

```csharp
// 规则 3：部分相关 → 推断并说明依据
// 规则 4：完全无关 → 明确拒绝
```

---

## 幻觉风险的真正边界

真正需要防范的幻觉是**第三格**：LLM 用它自己的训练数据编造了一个听起来合理但与用户笔记无关的答案。

现有的两层防幻觉体系针对的也正是这类情况：
1. **Answer Embedding Check**：对生成答案做 Embedding，与向量库比较，相似度过低则标记为潜在幻觉
2. **Self-Check Guard（可选）**：再调用一次 LLM 审核回答是否有文档依据

而"昨天买了三把，每天吃一把，推算明天还有"这类跨片段组合推理，不在幻觉防范范围内，是应该被允许的有益推理。

---

## 和 MCP / Agent 场景的区别

这个设计决策对**使用场景**非常敏感：

### 当前场景：直接面向用户的对话界面
- 用户期待有价值的回答，而不是原始 chunk
- **允许推理** 是正确选择
- System Prompt 应当中文、友好、鼓励推断

### 将来场景：作为 MCP Tool 供 Agent 调用
- 上层 Agent 有自己的推理层，期待 tool 返回**可信事实**，而不是已经加工过的推断
- RAG tool 应**回归严格模式**：只返回检索到的 chunks + 相似度分数，不经 LLM 生成
- 一个推理太多的 tool 会干扰 Agent 的决策链，造成"双重推理"污染

**结论**：同一个 RAG 后端在不同调用场景下应该有不同的行为模式。将来可通过 `mode` 参数（`conversational` vs. `retrieval-only`）在同一套代码里切换，不需要两套实现。

---

## 面试延伸点

- **RAG 的本质不是"搜索"，而是"检索增强的推理"**。严格限制推理是对 RAG 用途的误解。
- **Prompt Engineering 和业务场景强耦合**：同一个底层模型，不同 System Prompt 产生完全不同的用户体验。
- **防幻觉的正确姿势**：不是"禁止推理"，而是"标记不可信的推理"——让用户（或上层系统）自行决策是否采信。
- **面向 Agent 的 Tool 设计原则**：Tool 应该是原子的、确定性的、可验证的，把推理权交给 Agent，不要在 Tool 里做 LLM 推理。
