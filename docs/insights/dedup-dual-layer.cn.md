# 双层去重：哈希去重 + 向量相似度去重

**项目**：VedaAide.NET  
**涉及文件**：`src/Veda.Services/DocumentIngestService.cs`、`src/Veda.Storage/SqliteVectorStore.cs`  
**阶段**：Phase 2

---

## 问题背景

用户反复上传同一份或相似文档是常见场景（版本迭代、重复导入）。如果不做去重：
- 向量库里同一内容有多个副本，检索时被重复命中，回答里出现冗余引用
- 相似度分布被"撑高"，影响防幻觉校验的阈值判断
- 存储无限膨胀

---

## 为什么需要两层

### 第一层：哈希去重（完全相同）

对每个 chunk 的文本内容计算 SHA-256，存入数据库。摄取新 chunk 前先查哈希，命中则直接跳过。

**特点**：O(1) 查询，零额外成本，但只能识别**完全相同**的内容。

**局限**：文档里一个标点符号不同，哈希完全不同，无法识别"实质相同"的内容。

### 第二层：向量相似度去重（语义近似）

在 Embedding 生成之后、`UpsertBatchAsync` 之前，对每个新 chunk 做向量检索：

```csharp
// DocumentIngestService.cs
var similar = await vectorStore.SearchAsync(
    chunk.Embedding!, topK: 1, minSimilarity: dedupThreshold, ct: ct);
if (similar.Count == 0)
    deduped.Add(chunk);  // 无近似重复，保留
// 否则跳过，记录日志
```

`dedupThreshold` 默认 `0.95`（可通过 `appsettings.json` 调整），即余弦相似度 ≥ 0.95 视为语义重复。

**特点**：能识别改了几个词、换了说法但意思相同的内容，是真正的语义级去重。

---

## 两层的互补关系

```
新 chunk 到来
    │
    ├─ 哈希命中？ → 跳过（完全重复，O(1)，无 Embedding 开销）
    │
    └─ 哈希未命中 → 生成 Embedding → 向量检索
                        │
                        ├─ 相似度 ≥ 0.95？ → 跳过（语义重复）
                        └─ 相似度 < 0.95 → 存入向量库
```

哈希层是向量层的"廉价前置过滤"——完全相同的内容不需要浪费 Embedding 调用，直接在哈希层拦截。

---

## 阈值选择的权衡

`0.95` 是一个保守值，含义是"只过滤几乎一模一样的内容"：

- 阈值太高（如 `0.99`）：只过滤完全复制粘贴的内容，近似重复仍然入库
- 阈值太低（如 `0.80`））：同一话题的不同信息点被误判为重复，导致知识丢失

对于个人助理场景，`0.95` 是合理的下限。如果知识库里存的都是高度标准化的文档（如法规条文），可以适当降低到 `0.90`。

---

## 面试延伸点

- 为什么 RAG 系统需要去重：重复内容对检索排名和幻觉判断的干扰机制
- 哈希去重 vs. 向量去重的成本对比：一次 Embedding 调用的代价 vs. 一次哈希查询
- 去重阈值的业务含义：不是技术参数，而是"多像算重复"的业务决策
- 两层架构的设计模式：廉价守卫（cheap guard）在前，昂贵操作在后——一种通用的性能优化模式
